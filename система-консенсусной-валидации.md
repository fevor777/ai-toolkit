# Система консенсусной валидации

Мультиагентная система итеративного улучшения ответов через параллельную независимую проверку.

## Идея

Одна языковая модель может ошибиться, упустить важное или сгенерировать неточный ответ. Три независимых специализированных агента-критика, проверяющих ответ параллельно, значительно снижают эту вероятность — по принципу code review в разработке.

## Архитектура агентов

```
┌─────────────────────────────────────────────────────┐
│                   critic-router                      │
│           (оркестратор, до 2 раундов)                │
└─────────────────────┬───────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│                 critic-generator                     │
│     (генерирует черновик + собственную критику)      │
└─────────────────────┬───────────────────────────────┘
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
┌──────────────┐ ┌──────────┐ ┌────────────────┐
│    facts     │ │ clarity  │ │ completeness   │
│  (факты)     │ │ (ясность)│ │  (полнота)     │
└──────┬───────┘ └────┬─────┘ └───────┬────────┘
       └──────────────┴───────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│                   critic-router                      │
│         анализирует score, решает: готово?           │
└─────────────────────────────────────────────────────┘
```

## Процесс работы

```
Вопрос пользователя
       │
       ▼
  Раунд 1
       │
       ├─► critic-generator
       │         │
       │         ├─ черновик ответа
       │         └─ самокритика
       │
       ├─► [параллельно]
       │         ├─ critic-reviewer-facts       → score + BLOCKERS
       │         ├─ critic-reviewer-clarity     → score + BLOCKERS
       │         └─ critic-reviewer-completeness→ score + BLOCKERS
       │
       └─► critic-router считает средний score
                 │
          ┌──────┴──────┐
     score ≥ 8?        нет
          │              │
          ▼              ▼
    Финальный        Раунд 2
     ответ          (с учётом критики)
                         │
                         ▼
                    Финальный ответ
                    (с примечанием если score < 6)
```

## Критерии остановки

| Условие | Решение |
|---|---|
| Все три статуса `хорош` или средний score ≥ 8 | Завершить — вернуть финальный ответ |
| Есть `серьёзные проблемы` | Завершить с примечанием |
| score < 8 и раунд < 2 | Запустить следующий раунд |
| Раунд = 2 | Завершить в любом случае |

## Роли агентов

### critic-generator
Генерирует черновик ответа и сразу критикует себя — выявляет слабые места до того, как их найдут ревьюеры. При доработке получает сводную критику всех трёх ревьюеров и улучшает ответ.

### critic-reviewer-facts
Проверяет фактическую точность: нет ли ошибок, неподтверждённых утверждений, устаревших данных.

### critic-reviewer-clarity
Проверяет ясность и читаемость: логику изложения, структуру, понятность формулировок.

### critic-reviewer-completeness
Проверяет полноту: все ли важные аспекты раскрыты, нет ли пропущенных случаев.

### critic-router
Оркестратор: запускает агентов, собирает результаты, принимает решение о следующем раунде или финальном ответе.

## Аналогия с разработкой

```
Разработка ПО          Система консенсусной валидации
─────────────────────────────────────────────────────
Разработчик пишет код  critic-generator пишет черновик
Code review            critic-reviewer-* проверяют
Итерация по замечаниям Раунд 2 с учётом критики
Merge после апрува     Финальный ответ при score ≥ 8
```

## Применение

- Генерация текстов, планов, инструкций — там где важна точность и полнота
- Проверка кода — с заменой ревьюеров на специализированных (correctness, security, style)
- Любая задача где цена ошибки высока и нужна независимая верификация
